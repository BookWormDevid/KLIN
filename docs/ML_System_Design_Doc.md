## Термины и пояснения

> - Итерация — это все работы, которые совершаются до старта очередного пилота
> - БТ — бизнес-требования
> - EDA — Exploratory Data Analysis — исследовательский анализ данных
> - `Product Owner`, `Data Scientist` — роли, которые заполняют соответствующие разделы
> - В этом шаблоне роль `Data Scientist` совмещает в себе компетенции классического `Data Scientist` с упором на исследования и `ML Engineer` & `ML Ops` роли с акцентом на продуктивизацию моделей
> - Для вашей организации распределение ролей может быть уточнено в зависимости от операционной модели

## 1. Цели и предпосылки

### 1.1. Зачем идем в разработку продукта?

**Бизнес-цель `Product Owner`**
Создать браузерный продукт для автоматического обнаружения инцидентов на видео (насилие/драки/ДТП/взрывы/преступления против имущества), чтобы повысить безопасность на объектах и сократить время реакции служб безопасности и операторов.

**Почему станет лучше, чем сейчас, от использования ML `Product Owner` & `Data Scientist`**

- Снижение человеческого фактора: оператор не способен стабильно отсматривать весь поток видео без пропусков.
- Автоматическая нарезка и подсветка инцидентов сокращает время просмотра видео.
- Единая и воспроизводимая логика тревог вместо “субъективной оценки” операторов.
- Масштабирование мониторинга без линейного роста персонала.

**Что будем считать успехом итерации с точки зрения бизнеса `Product Owner`**
Итерация успешна, если:

- пользователь может загрузить видео/ссылку в браузере и получить результат (таймкоды + класс + визуальная подсветка bbox),
- система показывает метрики качества на пилоте (precision/recall/FPR),
- операторы дают позитивную обратную связь о полезности подсветки и нарезки событий,
- есть первые данные фидбека (подтверждение/отклонение тревог) для накопления датасета.

---

### 1.2. Бизнес-требования и ограничения

**Краткое описание БТ и ссылки на детальные документы с бизнес-требованиями `Product Owner`**
БТ (MVP):

- Работа в браузере без установки приложения.
- Вход: видеофайл и/или URL на видео | Потоковое видео с камеры
- Выход:
  - список найденных событий (таймкоды, тип аномалии, confidence),
  - клипы/превью по событиям,
  - визуализация зоны события через bbox (YOLO) на кадрах/в клипах,
  - выгрузка результата (JSON + клипы/видео с overlay — по возможности).
- Сбор аналитики по действиям пользователя и фидбеку (confirm/reject).

**Бизнес-ограничения `Product Owner`**

- Нет платных внешних источников данных.
- Ограниченные вычислительные ресурсы на старте.
- Сильно варьируется качество видео (освещение, угол, компрессия, FPS).
- Требуется быстрый MVP.

**Что мы ожидаем от конкретной итерации `Product Owner`**
Рабочий MVP детекции инцидентов с приемлемым качеством классификации и минимально необходимой визуализацией (bbox), пригодный для пилота на ограниченном наборе объектов/видео.

**Описание бизнес-процесса пилота `Product Owner`**

1) Пользователь загружает видео | вставляет URL | подключает камеру
2) Система выполняет обработку и находит события
3) Пользователь получает:
   - список событий с таймкодами и классом,
   - клипы/превью,
   - визуальную подсказку bbox (где происходило событие)
4) Пользователь подтверждает/отклоняет тревоги → сохраняем фидбек и улучшаем модель/порогование

**Что считаем успешным пилотом? Критерии успеха и возможные пути развития проекта `Product Owner`**
Целевые метрики пилота:

- Точность определения ≥ 85–90%
- Частота ложных тревог ≤ 10–15%
- Поддержка real-time/стриминга (24/7), алерты в системы инцидентов

Возможные пути развития:

- Улучшение доменной устойчивости (разные камеры/условия)
- Доразметка через human-in-the-loop и дообучение
- Доп. бизнес-логика приоритизации (P0/P1/P2) и эскалации событий

### 1.3. Что входит в скоуп проекта/итерации, что не входит

**На закрытие каких БТ подписываемся в данной итерации `Data Scientist`**

- 3-этапный ML-пайплайн:
  1) `X3D`: бинарная проверка видео (`аномалия/норма`),
  2) `VideoMAE`: определение типа аномалии только для видео/окон, где `X3D=аномалия`,
  3) `YOLO`: визуализация людей через **bbox** на кадрах с аномалией.
- Event-level результат: таймкоды событий (агрегация окон по времени).
- API-инференс + интеграция с web UI (upload/URL, получение результатов).
- Логирование фидбека операторов (confirm/reject).
- Потоковая обработка видео и алертинг.

**Что не будет закрыто `Data Scientist`**

- Массовое параллельное масштабирование без квот.
- Точный трекинг всех участников/оружия/объектов и точная пространственная локализация (сегментация/pose-level).
- Гарантированная “SOTA” точность на всех доменах без накопления собственного датасета.

**Описание результата с точки зрения качества кода и воспроизводимости решения `Data Scientist`**

- Версионирование датасета/сплитов и артефактов обучения (DVC/MLflow или аналог).
- Фиксированные конфиги обучения/инференса (yaml) и reproducible pipeline (скрипты/Makefile).
- Unit/интеграционные тесты для ключевых компонентов: decode, windowing, post-processing, API.
- Детерминированные сплиты train/val/test по видео/камерам.

**Описание планируемого технического долга `Data Scientist`**

- Оптимизация производительности (ONNX/TensorRT, батчирование, FPS downsample) — частично в MVP, углубление в следующих итерациях.
- Полноценный мониторинг качества в проде и data drift — после пилота.
- Расширенная визуализация (heatmap/attention) и трекинг (ByteTrack) — итерация 2.
- Автоматизация разметки и active learning — итерация 2–3.

---

### 1.4. Предпосылки решения

**Описание предпосылок `Data Scientist`**

- Используем видео (файлы/URL/Потоковое видео) как основной источник данных.
- Горизонт прогноза отсутствует: инференс выполняется по запросу, даже если выполняется real-time (batch).
- Гранулярность:
  - `X3D` делает бинарное предсказание (`аномалия/норма`) на окнах/клипах,
  - `VideoMAE` делает multi-class предсказание типа аномалии только если `X3D` сработал,
  - `YOLO` используется только для bbox-визуализации людей на кадрах с аномалией.
- Если `X3D` не сработал (`норма`), видео в пайплайне дальше не обрабатывается (`VideoMAE` и `YOLO` не запускаются).
- На уровне бизнеса события агрегируются в 3 группы:
  - `violent` — угроза здоровью/жизни,
  - `crimes` — угроза имуществу,
  - `Normal Videos` — не агрессия (без алерта).
- Предполагается наличие правового основания для обработки видео и соблюдение политики хранения/доступов.

## 2. Методология `Data Scientist`

### 2.1. Постановка задачи

**Что делаем с технической точки зрения `Data Scientist`**

- ML-задача состоит из 3 этапов:
  1) **X3D (binary classification)**: определить, есть ли аномалия/насилие на видео.
  2) **VideoMAE (multi-class classification)**: определить тип аномалии, если `X3D` сработал.
  3) **YOLO (spatial localization)**: выделить людей bbox на кадрах с аномалией для UI.
- Правило пайплайна: если `X3D=норма`, видео игнорируем; если `X3D=аномалия`, запускаем `VideoMAE`, затем `YOLO`.
- На выходе: список событий (таймкоды, класс, confidence) + визуализация bbox.

---

### 2.2. Блок-схема решения

**Блок-схема 3-этапного пайплайна (`X3D -> VideoMAE -> YOLO`) `Data Scientist`**

```text
[Upload/URL]
    |
[FFmpeg decode + normalize FPS/size]
    |
[X3D inference: anomaly / normal]
    |
  +------------------------------+
  | if normal -> stop/ignore     |
  | if anomaly -> continue       |
  +------------------------------+
               |
      [VideoMAE inference: anomaly-class probs]
               |
      [Temporal post-processing]
        - smoothing
        - hysteresis thresholds
        - merge windows -> events
               |
      [Business mapping]
        - Normal -> no alert
        - others -> violent/crimes
               |
      [YOLO bbox on anomaly frames (for visualization)]
    |
[Artifacts]
  - events.json
  - preview clips
  - frames/clips with bbox overlay
    |
[Web UI + feedback logging]
```

- [Пример возможной блок схемы](https://github.com/IrinaGoloshchapova/ml_system_design_doc_ru/blob/main/product_schema.png?raw=true)

### Этап 1 — Подготовка данных (EDA, разметка, сплиты)

#### Данные и сущности

- `video_id`, `source` (upload/url), `fps`, `resolution`, `duration`, `camera_id/location_id` (если доступно)
- `window_id/clip_id`, `t_start`, `t_end`
- `label_class_id` (0..13), `label_class_name`
- (опционально) `event_id` (если разметка event-level), `t_start`, `t_end`

#### Целевая переменная

- `y_x3d`: бинарная метка `anomaly / normal`
- `y_mae`: тип аномалии (multi-class, для окон/клипов с `y_x3d=anomaly`)
- На уровне бизнеса: `y_business_group`:
  - `violent` / `crimes` / `none` (для Normal)

#### Таблица: данные/источники

| Название данных                                    | Есть ли данные                                                 | Требуемый ресурс для получения данных (какие роли нужны) | Проверено ли качество данных (да, нет) |
| -------------------------------------------------- | -------------------------------------------------------------- | -------------------------------------------------------- | -------------------------------------- |
| Архив видеозаписей (пилот/история)                 | зависит от объектов/хранилища                                  | PO + Security + DS                                       | нет                                    |

#### Что делаем в EDA

- Проверили распределение 14 классов и долю `Normal` в локальной копии UCF-Crime.
- Проверили, как устроены готовые split-файлы (`Action_Regnition_splits`, `Anomaly_Detection_splits`).
- Проанализировали технические характеристики видео (`fps`, `resolution`, `duration`) через `ffprobe`.
- Проверили отсутствие утечек между train/test внутри каждого fold.

#### Итоги проведенного EDA (2026-03-03)

- Локально доступно `1950` видео:
  - `950` аномальных (13 классов),
  - `1000` normal (430 + 370 train normal, 150 testing normal, 50 normal для event recognition).
- По классам аномалий в локальной копии:
  - `RoadAccidents` и `Robbery`: по 150,
  - `Burglary` и `Stealing`: по 100,
  - остальные 9 классов: по 50.
- `Action_Regnition_splits` (4 fold) сбалансированы:
  - каждый `train_00i`: `532` видео (`38` на класс * 14 классов),
  - каждый `test_00i`: `168` видео (`12` на класс * 14 классов),
  - пересечение `train/test` внутри fold: `0`, пересечения между `test` fold: `0`.
- `Anomaly_Detection_splits`:
  - `Anomaly_Test.txt`: `290` видео, из них `150` normal (`51.7%`) и `140` anomaly (`48.3%`),
  - сильный дисбаланс внутри anomaly-классов: от `2` (`Abuse`) до `23` (`RoadAccidents`, `Shooting`),
  - `Anomaly_Train.txt` в текущей копии пустой (`0` строк) и не может использоваться как train-сплит.
- Качество видео на выборке `train_001 + test_001` (`700` видео):
  - `fps`: `29.97–30.0` (практически везде 30 FPS),
  - `resolution`: `320x240` для всех 700 видео,
  - длительность: min `3.467s`, median `75.302s`, mean `133.796s`, max `4730.02s`,
  - длинный хвост: `31%` роликов >= 120s, `7.43%` >= 300s.

#### Результат этапа

- Зафиксированы фактические характеристики датасета и сплитов для текущей локальной поставки.
- Подтвержден риск дисбаланса для anomaly detection (нужны `class weights` / `focal loss` / калибровка порогов).
- Подтвержден риск по длинам видео (нужны ограничение max duration и/или window sampling policy).
- Для воспроизводимости экспериментов в итерации MVP используем все `Action_Regnition_splits/train_00i` + `test_00i` в формате fold-by-fold (k-fold цикл), а итоговые метрики считаем как агрегат по fold.

### Этап 2 — Подготовка моделей X3D (бинарно) и VideoMAE (тип аномалии)

#### Бейзлайн

- `X3D` для бинарной классификации окон/клипов: `аномалия` / `норма`.
- `VideoMAE` для классификации типа аномалии на окнах, где `X3D=аномалия`.
- Простая схема окон (фиксированная длина клипа, фиксированный stride).
- Базовая логика: `X3D -> (if anomaly) VideoMAE -> post-processing`.

#### MVP (основной)

- Подбираем порог `X3D`, чтобы снизить пропуски аномалий (FN) при контролируемом FPR.
- Подбираем длину окна/stride под latency.
- Калибруем confidence `VideoMAE` (temperature scaling) для стабильных порогов.
- Улучшаем temporal post-processing (см. этап 4).

#### Формирование выборки (train/val/test)

- Сплит по видео/камерам/локациям.
- Для `X3D`: бинарные метки (`normal` vs `anomaly`).
- Для `VideoMAE`: обучение на аномальных классах (тип аномалии).
- Балансировка:
  - oversampling редких классов,
  - class weights или focal loss (если дисбаланс сильный).
- Augmentations (видео):
  - brightness/contrast/gamma, blur, JPEG artifacts,
  - random crop/resize,
  - time jitter (в пределах клипа), random frame drop.

#### Горизонт, гранулярность, частота пересчета

- Горизонт отсутствует; инференс по запросу.
- Гранулярность:
  - `X3D`: окно/клип -> `anomaly/normal`,
  - `VideoMAE`: окно/клип -> тип аномалии,
  - далее окно/клип -> событие.
- Пересчет: при каждой загрузке видео.

#### Метрики качества и связь с бизнесом

##### Все метрики

- Этап `X3D` (бинарный): precision/recall/F1 по классу `anomaly`, ROC-AUC, FPR.
- Этап `VideoMAE` (тип аномалии): macro-F1, weighted-F1, per-class precision/recall/F1, confusion matrix.
- Business-level:
  - метрики по `violent` vs `crimes` (агрегация классов),
  - отдельная метрика по “не агрессия” (Normal) как ключ к снижению FPR.
- End-to-end event-level (после постобработки):
  - event recall/precision,
  - частота ложных тревог (FPR) на N минут видео.

Связь:

- FPR определяет нагрузку на операторов (стоимость ложных тревог).
- Recall определяет риск пропуска опасного события (стоимость пропуска).

##### Основные метрики

- Precision - точность оценки моделью события (sum(TP) / sum(TP + FP))
- *Recall* - частота правильных срабатываний на реально происходящем событии (sum(TP) / sum(TP + FN))
- Accuracy - общая точность модели
- Для задачи самое важное - повышение точности модели при определении событий, снижение частоты ложных и увеличение частоты истинных срабатываний

#### Правила расчета event-level метрик

- Событие — непрерывный временной сегмент после temporal post-processing (smoothing + hysteresis + merge gap).
- Матчинг предикта и GT: temporal IoU >= tau (например 0.3) или пересечение >= t_min секунд; один GT матчится максимум с одним предиктом.
- `event precision/recall` считаются по совпавшим событиям.
- `FPR` считаем как число ложных событий на N минут/час видео.
- В критериях пилота "точность" = event precision, "ложные тревоги" = FPR на час видео.

#### Необходимый результат этапа

- Обученный и валидированный `X3D` для бинарного gate.
- Обученная и валидированная `VideoMAE` для типа аномалии.
- Зафиксированная логика запуска:
  - если `X3D=normal` -> видео игнорируем,
  - если `X3D=anomaly` -> запускаем `VideoMAE`.

#### Риски и план

- Дисбаланс/редкие классы → sampling/weights/active learning.
- Смешение смыслов классов → ревизия разметки + бизнес-агрегация.
- Доменные сдвиги → hard-set + аугментации.
- Ложные пропуски на `X3D` -> недетект аномалии в pipeline. Смягчение: порог на recall и ручной разбор FN.

#### Бизнес-проверка

- Топ-N ошибок по violent (особенно Shooting/Explosion/Fighting/Assault) — совместный просмотр с заказчиком/безопасностью.

### Этап 3 — Интерпретация моделей (согласование) и калибровка

#### Бейзлайн

- `X3D`: подбор бинарного порога, чтобы контролировать FN/FPR.
- `VideoMAE`: анализ confusion matrix (например Burglary vs Robbery vs Stealing).
- Базовая калибровка confidence для стабильного алерта.

#### MVP

- Раздельная калибровка по этапам:
  - `X3D`: рабочий порог запуска второго этапа (`VideoMAE`),
  - `VideoMAE`: пороги по бизнес-группам (`violent` строже, `crimes` мягче).
- Калибровка вероятностей (temperature scaling) для стабильной интерпретации confidence в UI.

#### Выход этапа

- Согласованные правила порогования для `X3D` и `VideoMAE` + карта классов → бизнес-группы.

---

### Этап 4 — Интеграция бизнес правил для расчета бизнес-метрик качества модели

#### Правила агрегации

- `X3D=normal` → `no alert`
- `X3D=anomaly` + класс `VideoMAE` → `alert` + `business_group`

#### Business mapping (фиксируем в коде и документации)

| Class                                                                | Business group |
| -------------------------------------------------------------------- | -------------- |
| Abuse, Arrest, Assault, Explosion, Fighting, RoadAccidents, Shooting | violent        |
| Arson, Burglary, Robbery, Shoplifting, Stealing, Vandalism           | crimes         |
| Normal Videos                                                        | none           |

#### Бизнес-метрики

- Precision/Recall/FPR отдельно по:
  - `violent` (priority high),
  - `crimes` (priority medium),
  - `none` (Normal) — ключевой для снижения ложных тревог.

#### Выход

- Отчет качества в терминах бизнеса + параметры порогов.

---

### Этап 5 — Подготовка инференса модели по итерациям

#### Бейзлайн

- CPU/GPU инференс `X3D -> VideoMAE` на клипах.
- Артефакты: events.json + preview.

#### MVP

- Оптимизация:
  - downsample FPS (например 8–12) с контролем качества,
  - батчирование клипов,
  - ONNXRuntime/torch.compile/TensorRT (при наличии GPU).
- Реализуем очереди задач, чтобы API не блокировался.

#### Выход

- Сервис инференса, интегрированный с UI и API.

---

### Этап 6 — Интеграция визуализации bbox (YOLO)

> Важно: YOLO используется **только** для bbox (визуализация “где”), не для определения класса.

#### Бейзлайн

- Детекция людей на кадрах внутри найденных событий, отрисовка bbox.

#### MVP

- Детекция людей + сглаживание bbox по времени (простое: median по координатам / NMS по соседним кадрам).
- Генерация overlay-клипов (или кадры с bbox) для UI.

#### Выход

- Визуализация зоны события для оператора.

---

### Этап 7 — Подготовка финального отчета для бизнеса

- Метрики per-class, по бизнес-группам и по главным метрикам.
- Кейсы ошибок (FP/FN) с примерами клипов.
- Рекомендации по следующей итерации: сбор данных, улучшение разметки, оптимизация.

## 3. Подготовка пилота

### 3.1. Способ оценки пилота

**Дизайн оценки `Product Owner`, `Data Scientist` c `AB Group`**

- **Offline:** прогон на размеченном наборе и расчёт метрик:
  - per-class,
  - `violent`/`crimes`,
  - FPR на минуту/час видео.
- **Online:** операторы подтверждают/отклоняют события в UI (human-in-the-loop).
- Если возможно A/B:
  - A: ручной просмотр без подсказок,
  - B: просмотр с подсказками (таймкоды + клипы + bbox).
- Единица оценки: event-level после post-processing; матчинг событий по правилам из раздела 2.3.
- Валидируем на holdout по камерам/локациям, чтобы снизить риск leakage.

---

### 3.2. Что считаем успешным пилотом

**Формализованные метрики `Product Owner`**

- Точность оценки ≥ 85–90%
- Ложно положительные события ≤ 10–15%
- Дополнительно: снижение времени просмотра исходного видео оператором (целевое значение фиксируется перед пилотом).
- Объем пилота (часы видео/число камер) фиксируется до старта, чтобы метрики были стабильны.
- Go/No-Go: пороги качества + устойчивость очередей и времени обработки в нагрузке.

---

### 3.3. Подготовка пилота

**Вычислительные ограничения и план расчетов `Data Scientist`**

- Замер на бейзлайне:
  - время обработки 1 минуты видео (end-to-end),
  - вклад `X3D` инференса,
  - вклад `VideoMAE` инференса,
  - вклад `YOLO` bbox-рендера,
  - размер артефактов (клипы/overlay).
- Ввести ограничения пилота:
  - max длительность видео (например до 10 минут),
  - max размер файла,
  - лимит параллельных задач (queue).
- Подготовить схемы артефактов (events.json, клипы/overlay) и хранение с ретеншеном.
- Включить логирование фидбека операторов (confirm/reject + причина).

---

### 3.4. Контур фидбека и артефакты пилота

**Что собираем и как используем `Data Scientist`**

- `events.json`: `event_id`, `t_start`, `t_end`, `class`, `confidence`, `business_group`, `source_id`.
- Feedback: `event_id`, `confirm/reject`, `reason`, `operator_id`, `timestamp`.
- Маппинг артефактов к исходному видео: `video_id` → клипы/overlay → фидбек.
- Регламент: выгрузка фидбека 1–2 раза в неделю для повторной разметки и дообучения.

---

## 4. Внедрение `для production систем, если требуется`

> На итерации 1 целимся в MVP-пилот. Раздел 4 заполняется, чтобы сразу зафиксировать путь production.

### 4.1. Архитектура решения

**Блок схема и пояснения: сервисы, назначения, методы API `Data Scientist`**

- Web UI: upload / URL / поток, список событий, просмотр клипов, фидбек
- Backend API:
  - upload / url fetch / stream fetch,
  - job queue,
  - storage (raw video + artifacts),
  - DB (метаданные + фидбек)
- Inference service:
  - X3D binary inference,
  - gate logic (`X3D=normal` -> stop; `X3D=anomaly` -> continue),
  - VideoMAE inference (тип аномалии),
  - temporal post-processing,
  - YOLO bbox inference (для визуализации),
  - artifact generator (clips/overlay/json)

Пример API:

- `POST /videos` → `job_id`
- `GET /jobs/{job_id}` → status
- `GET /videos/{video_id}/events` → events.json
- `GET /videos/{video_id}/artifacts` → clips/overlay
- `POST /events/{event_id}/feedback` → confirm/reject + comment

---

### 4.2. Описание инфраструктуры и масштабируемости

**Какая инфраструктура выбрана и почему `Data Scientist`**

- MVP: один inference-node (CPU или GPU) + очередь задач + объектное хранилище + streaming.
- Минимизация DevOps на старте, при этом возможность горизонтального масштабирования воркеров.

**Плюсы и минусы выбора `Data Scientist`**

- Плюсы: быстро, дешево, просто поддерживать.
- Минусы: ограничение по параллельности, риск очередей при пиковых загрузках.

**Почему финальный выбор лучше других альтернатив `Data Scientist`**

- Альтернатива (сложный real-real-time pipeline) дороже и не нужна для валидации бизнес-ценности на пилоте.

---

### 4.3. Требования к работе системы

**SLA, пропускная способность и задержка `Data Scientist`**

- SLA/latency фиксируются после замеров на пилотном железе:
  - target: обработка видео X минут ≤ Y минут end-to-end.
- Пропускная способность: ограничение N параллельных задач через queue.

---

### 4.4. Безопасность системы

**Потенциальная уязвимость системы `Data Scientist`**

- Вредоносные файлы/форматы → строгая валидация, ограничения размера, sandbox.
- DDoS/abuse загрузок → rate limits, auth, квоты.
- Утечки доступа к видео → RBAC, audit logs.

---

### 4.5. Безопасность данных

**Нет ли нарушений GDPR и других законов `Data Scientist`**

- Хранение видео по политике (сроки, доступы, удаление).
- Шифрование in-transit и at-rest.
- Минимизация доступа (least privilege).
- При необходимости: деперсонализация/маскирование лиц (в следующих итерациях, если потребуется).

---

### 4.6. Издержки

**Расчетные издержки на работу системы в месяц `Data Scientist`**

- Compute: X3D inference + VideoMAE inference + YOLO bbox + rendering
- Storage: raw video + clips/overlay
- Traffic: выдача клипов/overlay
- Labeling: часы разметки и QA (для итераций улучшения)

> Численные значения считаются по результатам замеров пилота.

---

### 4.7. Integration points

**Описание взаимодействия между сервисами (методы API и др.) `Data Scientist`**

- Интеграция с хранилищем видео (S3/локальное/корп. хранилище).
- Интеграция с системой инцидентов/алертов (опционально, следующая итерация).
- Экспорт результатов (JSON/events + ссылки на клипы) во внешние системы.

---

### 4.8. Риски

**Описание рисков и неопределенностей `Data Scientist`**

- Доменные сдвиги (другие камеры/условия) → hard-set, аугментации, дообучение.
- Дисбаланс классов и путаница между близкими классами → улучшение разметки, бизнес-агрегации, активное обучение.
- FPR выше целевого → пороги, hysteresis, min-duration, hard negatives.
- Перфоманс/стоимость → downsample FPS, батчи, оптимизация runtime.

---

### 4.9. MLOps и мониторинг качества

**Контур продуктивизации `Data Scientist`**

- Версионирование моделей/конфигов/данных, model registry и lineage.
- Деплой в reproducible контейнере, контроль совместимости инференса (CUDA/ONNXRuntime/torch).
- Мониторинг сервиса: p95 latency, глубина очереди, ошибки декодинга/инференса.
- Мониторинг качества: precision/recall на фидбеке, дрейф данных (fps/resolution/классы).
- Релизный контур: shadow/canary, критерии отката по SLO и качеству.
